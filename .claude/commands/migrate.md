description: Generate + apply an Alembic migration via the running backend container
argument-hint: "<migration message>"
---

Generate and apply an Alembic migration with message: **`$ARGUMENTS`**.

If `$ARGUMENTS` is empty or just whitespace, **stop** and ask the user for a meaningful migration message (e.g. `"add cheating_image_url to result"`). Don't proceed without one — empty messages produce migrations that are impossible to find later.

## Steps

1. Verify the backend container is running:
   ```bash
   docker ps --format '{{.Names}}' | grep -q '^nusmt_backend$'
   ```
   If it's not running, tell the user to run `make up` (dev) or `make prod-up` (prod) and stop.

2. Generate the migration **inside** the container (the container has the live DB connection + the latest model code):
   ```bash
   docker exec nusmt_backend sh -c "cd /face/app && uv run alembic revision --autogenerate -m '$ARGUMENTS'"
   ```

3. Copy the new migration file out of the container so it lives on the host (and gets committed to git):
   ```bash
   docker cp nusmt_backend:/face/app/migrations/versions/. ./backend/app/migrations/versions/
   ```

4. Apply the migration:
   ```bash
   docker exec nusmt_backend sh -c "cd /face/app && uv run alembic upgrade head"
   ```

5. Find the newly created migration file (the most recently modified `.py` under `backend/app/migrations/versions/`) and:
   - Print its filename.
   - Print the first 60 lines so the user can sanity-check the autogenerated upgrade/downgrade ops before committing.
   - Warn if any column drop or table drop appears — autogenerate sometimes proposes destructive ops the user didn't want.

## Constraints

- **Do NOT** run `alembic downgrade` or any destructive command without explicit user confirmation.
- **Do NOT** edit the generated migration unless the user asks — show it to them first.
- If `alembic revision --autogenerate` reports "Target database is not up to date", run `alembic upgrade head` first, then retry.
